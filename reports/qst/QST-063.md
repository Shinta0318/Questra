# QST-063 Report: Supabase Persistence Hardening

## Summary

Quest, Mission, Trail, Trail Reflection, Arc Memory, and Profile persistence boundaries were reviewed and hardened for Beta readiness. The main improvement is that save/load/delete failures no longer disappear silently for Quest and Mission flows, while Mission Arc Memory creation now follows successful Mission persistence.

## Changes

- Added shared `PersistenceSyncState` and `PersistenceSyncController`.
- Added `questSyncControllerProvider`.
- Added `missionSyncControllerProvider`.
- Updated Quest load/save/delete flows to surface loading, saved, and failed states.
- Updated Mission load/save flows to surface loading, saved, and failed states.
- Moved Mission Arc Memory extraction to run after Mission persistence succeeds.
- Updated Trail Event save failure handling so Mission sync state can surface the failure.
- Added `PersistenceSyncBanner` and wired it into Quest and Mission screens.
- Updated Profile onboarding persistence to restore prior profile state and expose `AuthState.errorMessage` on failure.
- Added unit coverage for persistence sync state.

## Persistence Boundaries

- Quest
  - Create/update: `QuestController._persistQuest -> QuestRepository.save(ownerId, quest)`
  - Fetch: `QuestController.loadForUser -> QuestRepository.findByUser(userId)`
  - Delete: `QuestController._deleteQuest -> QuestRepository.delete(ownerId, questId)`
  - Owner boundary: `owner_id`

- Mission
  - Create/update: `MissionController._persistMission -> MissionRepository.save(mission)`
  - Fetch: `MissionController.loadForQuests -> MissionRepository.findManyByQuestIds(questIds)`
  - Owner boundary: RLS derives owner through parent Quest.

- Trail / Trail Reflection
  - Create/update: `TrailController._persistTrail -> TrailRepository.save(ownerId, trail)`
  - Reflection: `TrailController.updateTrail` persists `TrailType.arcReflection`
  - Owner boundary: `owner_id`

- Arc Memory
  - Quest: saved with `questId`
  - Mission: saved with `questId` and `missionId` after Mission persistence succeeds
  - Trail: saved with `questId`, `missionId`, and `trailId`
  - Owner boundary: `user_id`

- Profile/User
  - Signup/login profile upsert/load remain Supabase-backed.
  - Onboarding profile persistence now reports failure through `AuthState`.

## Migration Candidates

No blocking migration is required for the current MVP schema, but these columns should be considered before Beta if the app should preserve all model fields remotely:

- `quests.progress numeric not null default 0`
- `quests.category text not null default '冒険'`

Optional query improvement:

- Mission list screens currently fall back to a local/default Quest title when loading from Supabase because `missions` does not store `quest_title`. Prefer a joined query/view for display rather than duplicating the title.

## Acceptance

- Quest persistence status is visible to the user.
- Mission persistence status is visible to the user.
- Trail and Trail Reflection continue to use sync status banners.
- Arc Memory is linked to Quest, Mission, and Trail identifiers where available.
- Profile persistence failure is surfaced.
- RLS ownership assumptions remain aligned with the current schema.
